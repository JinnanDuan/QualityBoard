# 角色设定

你是本项目的产品设计专家，精通数据看板与仪表盘设计。你擅长在首页合理布局「概览卡片」与「趋势图表」，兼顾信息密度与可读性。你熟悉 ECharts 折线图、双 Y 轴、Tooltip 等常见图表交互设计。

# 任务目标

请输出一份 Markdown 格式的 **首页最新批次状态 + 趋势折线图功能规约（Spec）**，不要写任何代码。该规约将作为后续 AI 编程的契约，用于实现该功能。本次仅输出 Spec，不做代码实现。

# 输入信息

## 1. 功能概述

在系统首页（Dashboard，路由 `/`）展示：
- **最新批次状态**：以卡片或摘要形式展示最近一个批次的执行概况
- **趋势折线图**：以批次为横轴的折线图，展示多批次执行趋势

## 2. 数据源：pipeline_overview 表

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | int(11) PK | 自增主键 |
| batch | varchar(100) | 轮次/批次标识 |
| subtask | varchar(100) | 组别（一个机器一个组） |
| result | varchar(25) | 本轮该组执行结果（passed/failed） |
| case_num | varchar(25) | 本轮该组执行的所有用例数量 |
| batch_start | datetime | 本轮该组开始执行时间 |
| batch_end | datetime | 本轮该组执行结束时间 |
| passed_num | int(11) | 本轮该组执行通过的用例数量 |
| failed_num | int(11) | 本轮该组未执行通过的用例数量 |
| platform | varchar(255) | 平台名称 |
| code_branch | varchar(255) | 代码分支 |
| reports_url / log_url / screenshot_url / pipeline_url | varchar | 各类 URL（本次可不展示） |

**说明：** 同一 `batch` 下可能有多条记录（不同 subtask/platform），聚合时需按 `batch` 分组。

## 3. PRD 参考（Story 1.1 批次执行概览趋势图）

- **AC 要点：**
  - X 轴：批次（按自增顺序自左向右排列）
  - 趋势线：失败用例数、总用例数、执行耗时/执行时间
  - Tooltip：悬停时展示该批次完整数据（执行时间、成功数、未处理数、执行耗时）
  - 默认展示最近 N 个（如 30 个）批次，避免 X 轴拥挤

- **数据聚合规则：** 按 `batch` 聚合：`SUM(case_num)` 总用例数、`SUM(failed_num)` 失败数、`SUM(passed_num)` 通过数；执行耗时 = `MAX(batch_end) - MIN(batch_start)`。

## 4. 现有实现

- **首页：** `frontend/src/pages/dashboard/DashboardPage.tsx`，当前为占位内容「首页大盘」
- **技术栈：** React 18、TypeScript、Ant Design 5、ECharts
- **路由：** `/` 对应 DashboardPage

## 5. 业务术语

| 术语 | 含义 |
|------|------|
| 批次 (Batch) | 一轮完整的测试执行，对应 `pipeline_overview.batch` |
| 分组 (Subtask) | 批次下按机器/任务划分的执行单元 |
| 最新批次 | 按 `batch_start` 或 `batch` 排序取最近的一条（或聚合后最近的一条） |

# 规约必须明确界定的内容

请对以下两部分分别输出详细规约：

## 一、最新批次状态卡片

1. **展示内容**：明确展示哪些指标（如：批次、总用例数、通过数、失败数、通过率、执行耗时、执行时间范围等）
2. **布局与样式**：卡片数量、排列方式（横向/纵向）、每项指标的展示形式（数字、标签、进度条等）
3. **数据聚合规则**：最新批次如何定义（按 batch_start 取最新？按 batch 字典序？）；同一 batch 下多 subtask 如何聚合（SUM / 取第一条 / 其他）
4. **空数据与异常**：无数据时展示策略；数据异常（如 case_num 为空）时的兜底展示

## 二、趋势折线图

1. **X 轴**：批次展示方式（批次标识、排序规则、展示数量 N 的默认值及可配置性）
2. **Y 轴与指标**：明确每条折线/柱状图对应的指标（失败数、总用例数、执行耗时等），以及是否采用双 Y 轴（若指标量纲差异大）
3. **图表类型**：纯折线 / 折线+柱状组合 / 其他，需说明理由
4. **Tooltip**：悬停时展示的完整字段列表及格式
5. **交互**：点击数据点是否跳转（如跳转到 History 页并带 batch 筛选参数）；是否支持图例切换显示/隐藏
6. **数据范围**：默认查询最近 N 个批次；是否支持用户切换时间范围或批次数量

## 三、可选扩展（规约中说明是否纳入）

- 是否支持按平台（platform）或代码分支（code_branch）筛选展示？
- 最新批次卡片是否可点击跳转到详情页？

## 四、数据获取与 API 设计

1. **后端 API**：需要几个接口？每个接口的请求参数、响应结构（字段名、类型、说明）
2. **聚合逻辑**：在 Service 层按 batch 聚合的 SQL 或伪代码逻辑
3. **性能考虑**：pipeline_overview 表数据量级；是否需要限制查询批次数量、索引建议
