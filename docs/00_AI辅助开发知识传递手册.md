## 一、From the Ground Up：从 0 到 1 搭建脚手架

### 1.1 ORM 模型与数据结构绑定

**精读文档**：`database/README.md`

**核心思路**：项目从 0 到 1 搭建时，**首先完成**该文档的编写。通过该文档了解**跨区开发**的思路。

**要点摘要**：

| 维度 | 说明 |
|------|------|
| **SQL 是唯一真理** | 严禁依赖 ORM 自动生成/修改表结构；所有 DDL/DML 必须以 SQL 脚本形式留存；ORM 实体必须根据 SQL 文件**反向同步**编写 |
| **MySQL 5.7 锁定** | 禁止使用 MySQL 8.0+ 特性；字符集 `utf8mb4`，排序规则 `utf8mb4_unicode_ci` |
| **命名规范** | `V<主版本号>.<次版本号>.<修订号>__<全英文下划线描述>.sql`（版本号与描述间为**双下划线**） |
| **跨网络流转** | 开发环境（蓝区）编写 SQL → 本地执行 → 同步 ORM → 业务开发；生产（黄区）按版本号升序执行新脚本 |

## 二、某特性的 AI 辅助开发思路

### 2.1 拆解并挑选起点

- 将 feature 尽可能拆到**最小**
- 挑选一个**最小的 feature** 去完成开发

---

### 2.2 Prompt 生成（spec-prompt-gen）

| 步骤 | 说明 |
|------|------|
| **2.2.1** | 将上述最小的 feature 用**自然语言**描述，把自己想到的**关键点**都写出来 |
| **2.2.2** | 将描述给到 AI，让 AI 辅助生成**用于生成 spec 的 prompt** |
| **2.2.3** | **明确提示 AI**：不是直接生成 spec，更不要开始开发，而是**只生成用于生成 spec 的 prompt** |
| **2.2.4** | **人工对 prompt 进行完整的审阅和修改** |

---

### 2.3 Spec 生成（spec-gen）

| 步骤 | 说明 |
|------|------|
| **2.3.1** | 将上述生成的 spec-prompt 给到 AI，并给到指令：**只生成 spec，不进行代码开发** |
| **2.3.2** | **人工对 spec 进行完整的审阅和修改** |

---

### 2.4 代码生成（spec-dev）

| 步骤 | 说明 |
|------|------|
| **2.4.1** | 将确认后的 spec **直接给到 AI** |
| **2.4.2** | 进行**代码 review、调试、调整** |

---

## 五、Cursor 用量注意点

### 5.1 配额

- 每月约 **20 刀** 额度
- **自选高阶模型**：配额消耗很快
- **Auto 模式或 Composer 1.5 模型**：配额相对多一些

### 5.2 使用策略

| 策略 | 说明 |
|------|------|
| **1** | 日常一般选定 **Composer 1.5** 使用，效果不错，基本 feat 开发和 bugfix 在 1～2 次可解决 |
| **2** | 一般不要选定其他模型使用；若 Composer 1.5 搞不定，可考虑使用 **Opus 4.6** |
| **3** | 与项目无关的问答，**不要使用 Cursor**，尽量节省 token |
