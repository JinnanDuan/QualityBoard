# DT 看板开发任务拆解与计划倒排

> **定位说明**：本系统是【自动化测试一轮跑完后的作战看板】，不是完整功能型测试平台，也不是 BI/报告系统/企业后台。  
> 本文档按**真实使用流程**梳理需求优先级、模块拆分与实现顺序。

---

## 〇、真实使用流程（核心主路径）

用户每天使用系统的固定流程：

```
1. 打开 DT 看板首页
   → 只看最近一轮已跑完的「最新批次」
   → 通过趋势图/统计卡片快速确认这一轮的状态

2. 点击该批次，自动进入 History 页面
   → 自动带入 batch / branch / platform 等筛选条件
   → 不需要手动再选一遍

3. 在该批次下进行失败处理
   → 自动继承历史失败原因（上一轮已标注的，本轮自动带出）
   → 剩余未继承到原因的失败用例 = 新增失败

4. 对新增失败用例进行批量操作
   → 一键分析
   → 一键标记为「bug」

5. 一轮处理完成后，生成【总结报告】
   → 这是「每一轮跑完必须发群里的总结」
   → 不是 BI 报告，也不是事后复盘

6. 基于该轮总结结果
   → 把「失败原因 = bug」的用例做成文本总结
   → 直接复制到群里
   → 并给这些失败用例的跟踪人一键发送 WeLink 消息
```

## 一、功能优先级重排（按真实使用频率）

| 优先级 | 功能 | 使用频率 | 说明 |
|--------|------|----------|------|
| **P0** | 首页 → 最新批次状态 | 每轮必用 | 统计卡片 + 趋势图，一眼看清本轮状态 |
| **P0** | 点击批次 → History 自动带参 | 每轮必用 | 无需手动选 batch/branch/platform |
| **P0** | 失败原因继承 | 每轮必用 | 历史已标注的自动带出，未继承 = 新增失败 |
| **P0** | 批量标注（一键分析/一键 bug） | 每轮必用 | 对新增失败做批量操作 |
| **P0** | 总结报告（群发文本 + 一键复制） | 每轮必用 | 作战工具，不是汇报用 |
| **P0** | 基于批次的批量 WeLink 通知 | 每轮必用 | 总结后一键通知所有 bug 跟踪人 |
| **P1** | 流转指派时通知下一处理人 | 流转时 | 失败原因 bug→bug 且 owner 变更时，给新 owner 发 WeLink（先预留 API 调用） |
| **P1** | 操作时间线 | 偶尔查看 | Drawer 内 owner_history 展示 |
| **P1** | 失败类型字典维护 | 初始化/偶尔 | 管理员维护 bug 等类型，可先写死 |
| **P2** | 用户/模块映射管理 | 初始化/偶尔 | 支撑指派、模块默认值 |
| **P2** | 定时催办、防打扰 | 可选 | 非每轮必需 |
| **P3** | 审计日志、复杂报告样式、多批次对比 | 后置 | 不优先 |

## 二、Report 模块定位与拆分

### 2.1 报告模块 ≠ 汇报用，报告模块 = 作战工具

**Report MVP 目标**：让我不用再手工整理总结内容。

### 2.2 Report MVP 版本（必须）

| 能力 | 说明 |
|------|------|
| 固定一个批次 | 选择当前轮批次（或默认最新批次） |
| 自动统计 | 总用例数、成功数、失败数 |
| 失败原因 = bug 的用例 | 按 platform 区分，按跟踪人分组，**按模块统计数量**（不列出具体用例名） |
| 输出【群可直接发送的文本总结】 | 固定格式，见 2.4 |
| 一键复制 | 点击复制到剪贴板，直接粘贴到群 |
| 详见链接 | 带 `batch=xx&failed_type=bug` 的 History 筛选 URL |

**MVP 不包含**：复杂图表、多批次历史对比、精美报告样式、分享链接、深度链接。

### 2.4 总结文本格式（固定）

```
【rolling线看护进展通告】
batch：{batch}
构建分支：{code_branch}

用例总数：xx，成功：xx，失败：xx

怀疑修改引入问题：
oh：
  小明 123456 【1条】：plugin（1）
  小红 123123 【2条】：output（1），git（1）
（按 platform 区分，每个 platform 一块；每块内按跟踪人：姓名 工号 【X条】：模块1（数量），模块2（数量）；不列出具体用例名）

详见：{base_url}/history?start_time={batch}&failed_type=bug
```

- **batch**：批次（如 start_time）
- **构建分支**：code_branch
- **按 platform 区分**：不同 platform（如 oh、android 等）各一块，platform 名作为小标题
- **按跟踪人统计**：每人一行，格式为 `姓名 工号 【X条】：模块A（数量），模块B（数量）`
- **不列出具体用例名**：仅按模块统计失败用例数量

### 2.3 Report 后续增强版本（可选）

| 能力 | 说明 | 优先级 |
|------|------|--------|
| 分享链接 | 生成唯一 URL，他人可查看 | P2 |
| 图表点击跳转 | 点击「Bug 10 条」→ 打开 History 并带筛选 | P2 |
| 报告快照 | 生成时落盘，防止后续数据变更影响 | P2 |
| 饼图/分析完成度 | 可视化展示 | P3 |

## 三、通知模块最小实现方案

### 3.1 真实用法：总结后统一触发，非事件驱动

- **不是**：每次指派就通知
- **而是**：一轮处理完 → 基于该轮 bug 失败用例 → 一键通知所有跟踪人

### 3.2 流转指派通知（补充）

开发人员在处理自己的失败用例时，可能需将用例流转给其他模块或其他开发进行分析。**当流转发生时**（如失败原因保持 bug→bug，但 owner 变更），需给**下一个处理人**发送 WeLink 消息通知。

- **实现策略**：在 `process_failure` 中，当 owner 发生变更时，预留通知 API 调用（如 `notification_service.notify_assignee(employee_id, ...)`），后续接入 WeLink 真实发送。
- **当前**：先预留调用点，不阻塞主流程。

### 3.3 最小实现方案

| 能力 | 说明 |
|------|------|
| WeLink 客户端 | 封装 WeLink API，发送文本/卡片消息 |
| 基于【批次 + 条件】的批量通知 | 选择批次 + 条件（如 `failed_type=bug`）→ 按跟踪人分组 → 每人一条合并消息：「您有 X 条 Bug 失败用例待跟进：[链接]」 |
| 流转指派通知 | owner 变更时预留通知 API 调用，后续接入 WeLink |
| 入口 | 在 Report 页面或 History 页面提供「一键通知 Bug 跟踪人」按钮 |
| 校验 | 若所选条件下无待处理用例，阻断并提示 |

**最小方案不包含**：定时催办、防打扰合并。这些可后续按需增加。

## 四、按真实工作流划分的开发阶段

### 4.1 阶段总览

```
阶段 1：首页 → 批次 → History 闭环
阶段 2：失败处理增强（继承 + 批量）
阶段 3：总结报告 MVP
阶段 4：批量 WeLink 通知
阶段 5：管理后台（按需最小化）
```

### 4.2 阶段 1：首页 → 批次 → History 闭环

**目标**：打开首页看最新批次，点击即进入 History 且自动带参。

| 任务 ID | 任务名称 | 工作量 | 产出 |
|---------|----------|--------|------|
| S1-1 | Dashboard 最新批次统计 API | 0.5d | `GET /dashboard/stats`：最新批次、总/通过/失败/通过率、未处理数、分析完成度 |
| S1-2 | Dashboard 趋势图 API（可选简化） | 0.5d | 最近 N 批趋势，或首版仅统计卡片 |
| S1-3 | Dashboard 前端 | 1d | 统计卡片 + 可选简单趋势图，点击「最新批次」跳转 History |
| S1-4 | History 支持 URL 参数自动筛选 | 0.5d | `?batch=xxx&branch=xxx&platform=xxx` 自动填入并触发查询 |
| S1-5 | 首页「进入本批次」按钮 | 0.5d | 直接带最新批次参数跳转 `/history?start_time=xxx` |

**交付物**：首页可看最新批次状态，点击即进 History 且无需手动选筛选条件。

---

### 4.3 阶段 2：失败处理增强（继承 + 批量）

**目标**：自动继承历史失败原因，对新增失败支持一键分析、一键标记 bug。

| 任务 ID | 任务名称 | 工作量 | 产出 |
|---------|----------|--------|------|
| S2-1 | 失败原因继承逻辑 | 1d | 当前批次失败用例：查上一批次同 case_name+platform 是否有 pipeline_failure_reason，有则自动 INSERT/UPDATE 到当前批次，并标记为「已继承」 |
| S2-2 | History 列表区分「新增失败」 | 0.5d | 新增失败 = 当前批次 failed 且无 pipeline_failure_reason 记录，可选列或筛选 |
| S2-3 | 一键分析（批量同类型标注） | 0.5d | 勾选多条 → 选失败类型+跟踪人 → 批量提交，复用现有 process_failure |
| S2-4 | 一键标记为 Bug | 0.5d | 快捷按钮，等价于批量选 failed_type=bug + 模块默认值 |
| S2-5 | 流转指派时预留通知 API 调用 | 0.5d | process_failure 中 owner 变更时调用 `notify_assignee(new_owner_employee_id, ...)`，先占位不真实发送 |
| S2-6 | 操作时间线（可选） | 0.5d | Drawer 内解析 owner_history 按时间倒序展示 |

**交付物**：进入批次后，历史原因自动带出，新增失败可批量标注、一键 bug；流转时预留通知调用点。

---

### 4.4 阶段 3：总结报告 MVP

**目标**：固定批次 → 自动统计 → 输出群发文本 → 一键复制。

| 任务 ID | 任务名称 | 工作量 | 产出 |
|---------|----------|--------|------|
| S3-1 | Report 统计 Service | 1d | 输入批次 → 总/成功/失败数；failed_type=bug 的用例按 platform、owner、main_module 聚合（每人每模块数量） |
| S3-2 | Report 文本模板 | 0.5d | 按 2.4 固定格式：platform 分块 → 每块内按跟踪人 → 每人「姓名 工号 【X条】：模块（数量）」 |
| S3-3 | Report API | 0.5d | `GET /report/summary?batch=xxx` 返回统计 + 文本 |
| S3-4 | Report 前端 | 1d | 选择/默认批次 → 展示统计 + 文本区 → 一键复制按钮 |

**交付物**：不用手工整理，直接复制到群里。

---

### 4.5 阶段 4：批量 WeLink 通知

**目标**：基于批次 + 条件（如 bug）→ 一键通知所有跟踪人。

| 任务 ID | 任务名称 | 工作量 | 产出 |
|---------|----------|--------|------|
| S4-1 | WeLink 客户端封装 | 1d | 调用 WeLink API 发送消息，接收人用 ums_email.domain_account |
| S4-2 | 批量通知 Service | 1d | 输入批次 + failed_type → 按 owner 分组 → 每人一条合并消息，含待跟进数量 + 直达链接 |
| S4-3 | 通知 API | 0.5d | `POST /notification/batch`，校验有待处理用例才发送 |
| S4-4 | Report 页「一键通知 Bug 跟踪人」 | 0.5d | 按钮调用 API，成功/失败提示 |

**交付物**：总结后一键发 WeLink 给所有 bug 跟踪人。

---

### 4.6 阶段 5：管理后台（按需最小化）

**原则**：不为了「系统完整性」强行提前。优先解决每天是否还需手工做重复工作。

| 任务 ID | 任务名称 | 工作量 | 说明 |
|---------|----------|--------|------|
| S5-1 | 失败类型字典 CRUD | 0.5d | 支撑「一键 bug」等，可先写死几条 |
| S5-2 | 用户管理（ums_email） | 0.5d | 支撑指派、通知，可先导入/简单维护 |
| S5-3 | 模块映射（ums_module_owner） | 0.5d | 支撑 bug 默认跟踪人 |

**后置**：用例上下线、下线类型字典、定时催办配置、审计日志、通知渠道配置。

## 六、开发阶段与排期建议

| 阶段 | 目标 | 预估 | 累计 |
|------|------|------|------|
| 阶段 1 | 首页 → 批次 → History 闭环 | 3d | 3d |
| 阶段 2 | 失败处理增强（继承 + 批量） | 3d | 6d |
| 阶段 3 | 总结报告 MVP | 3d | 9d |
| 阶段 4 | 批量 WeLink 通知 | 3d | 12d |
| 阶段 5 | 管理后台最小化 | 1.5d | 13.5d |

**MVP 可交付节点**：阶段 1～4 完成后，即可支撑「每轮跑完 → 看板 → 处理 → 总结 → 通知」全流程，无需手工整理。
