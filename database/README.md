# 数据库迁移与版本控制规范 (Database Migration Guide)

本项目采用严格的纯 SQL 文件进行数据库结构版本控制。

由于我们的开发环境与生产环境处于**物理隔离的网络区域**，且必须保证双端数据库结构的绝对一致性，请所有参与开发的工程师（以及辅助开发的 AI 助手）严格遵守以下规范。

## ⚠️ 核心原则 (Core Principles)

1. **SQL 是唯一的真理 (Single Source of Truth)**
   - 严禁完全依赖框架级的 ORM 工具（如 SQLAlchemy, TypeORM, GORM, Prisma, MyBatis-Plus 等）自动生成或修改线上表结构。
   - 所有的 DDL（建库、建表、改字段、加索引）和基础字典数据的 DML（初始化配置数据），**必须**以 SQL 脚本的形式留存在本目录下。
   - 后端代码中的 ORM 实体类/数据模型文件，必须根据本目录下的 SQL 文件反向同步编写，保持严格一致。

2. **严格锁定 MySQL 5.7 版本 (Version Lock)**
   - 生产环境运行的是 MySQL 5.7，开发环境必须保持一致。
   - **禁止**在 SQL 脚本中使用任何 MySQL 8.0+ 的专属特性（如：隐藏索引、`CHECK` 约束、`utf8mb4_0900_ai_ci` 排序规则、`WITH` 通用表表达式等）。
   - 默认字符集一律使用 `utf8mb4`，排序规则统一使用 `utf8mb4_unicode_ci`。

## 📁 命名规范 (Naming Convention)

所有新增的 SQL 文件必须严格遵循以下 Flyway 风格的命名格式：

`V<主版本号>.<次版本号>.<修订号>__<全英文下划线描述>.sql`
*(注意：版本号与描述之间是**双下划线 `__`**)*

**正确示例：**
- `V1.0.0__init_admin_schema.sql` (初始化建库)
- `V1.0.1__create_pipeline_history.sql` (新建流水线历史表)
- `V1.0.2__add_status_to_history.sql` (为历史表增加状态字段)
- `V1.1.0__insert_init_admin_user.sql` (插入初始管理员账号数据)

## 🔄 跨网络开发与部署流转 (Workflow)

鉴于黄区（生产）与蓝区（开发）无法直连，请按照以下标准流程操作：

### 在开发环境（蓝区）
1. 有新的数据库需求时，在 `database/` 目录下创建新的 `Vxxx__xxx.sql` 文件并编写 SQL。
2. 在本地开发库（蓝区MySQL）中手动执行该 SQL。
3. 同步修改后端代码中的 ORM 实体模型。
4. 编写业务逻辑代码并进行功能测试。

### 向生产环境迁移（黄区）
1. 将包含最新 SQL 文件的代码包拷贝至黄区。
2. DBA 或部署人员**按版本号升序**，在生产库（黄区MySQL）中依次执行尚未执行过的新 SQL 脚本。
   *(例如：生产库目前结构停留在 V1.0.1，本次发布包含了 V1.0.2 和 V1.1.0，则只需按顺序执行这两个新脚本)*
3. 确认数据库无误后，启动/重启后端 Web 服务。

## 🤖 AI 辅助开发须知 (For AI Coding Assistants)

如果使用 Cursor、Copilot 等 AI 工具辅助编写数据库相关代码，请务必在 Prompt 中强调本规范：
> "本项目数据库使用 MySQL 5.7，请使用标准的 5.7 SQL 语法，字符集使用 utf8mb4。不要依赖特定语言的 ORM 自动迁移工具生成表结构，请直接为我生成符合 Vxxx__xxx.sql 命名规范的 DDL 脚本文件。"
