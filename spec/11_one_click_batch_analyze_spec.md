# 一键分析（整批 Bug）功能规约（Spec）

本文档在 `spec/02_history_fields_spec.md`、`spec/03_history_failure_reason_spec.md`、`spec/04_failure_process_spec.md`、`spec/10_inherit_failure_reason_spec.md` 基础上，定义「一键分析」的交互、数据写入与 API 规约。

**文档位置说明**：本功能为独立能力（**新接口**、与「分析处理」失败标注分流），故单独成文；与「失败原因继承」同属批量类能力，编号顺延为 `11`。

---

## 1. 功能概述

### 1.1 功能目标

在**同一批次（轮次）内，对当前仍未分析**的失败/异常用例，**一键**批量写入失败原因：**失败类型 = bug**，**跟踪人 = 该用例的用例开发责任人**（存储格式为 **姓名+工号**），**详细原因**为统一模板文案，**分析人**为当前登录用户，**分析时间**以写入时刻为准（见 `pipeline_failure_reason.updated_at`）。

### 1.2 与「分析处理」（失败标注）的区别


| 维度                        | 分析处理（`spec/04`）                           | 一键分析（本文档）                                                                        |
| ------------------------- | ----------------------------------------- | -------------------------------------------------------------------------------- |
| 接口                        | 现有 `POST /api/v1/history/failure-process` | **独立新接口**（见第 7 节）                                                                |
| 作用范围                      | 用户勾选的若干条 `history_id`                     | **整批**：与锚点相同 `start_time` 的全表记录（不按 `subtask` 切分）                                 |
| 失败类型                      | 用户选择                                      | 固定为 **bug**（见第 5.2 节字典值）                                                         |
| 跟踪人                       | 用户选择，提交值为工号                               | **按用例解析**：`main_module` → `ums_module_owner`（必要时 `ums_email`），写入 **「姓名+工号」** 字符串 |
| 模块                        | bug 时须选手动模块                               | **不选模块**；不按「非 main_module 责任」场景处理，直接按用例开发责任人写入                                   |
| `pipeline_failure_reason` | 含 `owner`、`failed_type`、`reason` 等        | **无 `module` 字段**（表结构本身无此列）；不额外写入模块                                              |


### 1.3 适用页面

- **详细执行历史**页面（与「分析处理」「继承失败原因」一致）。

---

## 2. 作用范围与批次界定

### 2.1 整批定义

- **不按 `subtask` 切分**。
- 以锚点记录的 `**pipeline_history.start_time`** 作为**唯一批次键**，查询**全库**中满足下列条件的所有 `pipeline_history` 行（不限 `subtask`）：

### 2.2 目标记录条件（必须同时满足）


| 条件            | 说明                                          |
| ------------- | ------------------------------------------- |
| `start_time`  | 等于锚点记录的 `start_time`                        |
| `case_result` | `'failed'` 或 `'error'`（与现有失败标注一致）           |
| `analyzed`    | `**0` 或 `NULL`**（与 **失败原因继承**目标筛选一致：仅未分析用例） |


### 2.3 排除

- `**analyzed = 1`** 的记录：不参与一键分析（既不更新也不计数为成功对象）。

---

## 3. 入口与交互

### 3.1 操作路径

1. 用户勾选**任意一条**非 passed 用例（即 `failed` 或 `error`）。
2. 点击 **「一键分析」** 按钮（具体文案以 UI 为准，含义为本功能）。
3. 前端提交 `**anchor_history_id`**（见第 7 节），后端按第 2 节解析批次并执行。

### 3.2 按钮可用条件（建议）


| 状态  | 条件                                                  |
| --- | --------------------------------------------------- |
| 可用  | 至少勾选 **1 条** `case_result` 为 `failed` 或 `error` 的记录 |
| 禁用  | 未勾选，或勾选的全为 `passed`                                 |


### 3.3 锚点 `history_id` 的选取

- 请求体携带 `**anchor_history_id`**。
- 若用户勾选多条，前端任选**其中一条**的 `id` 作为锚点即可（建议取勾选集合中的第一条，避免歧义）。
- 后端**不以**勾选列表为作用范围，**仅以**锚点解析出的 `start_time` 决定整批（见第 2 节）。

---

## 4. 锚点校验

后端应对 `anchor_history_id` 做如下校验，不通过则返回 **400** 及明确中文 `detail`：


| 校验项                                | 失败提示（示例）          |
| ---------------------------------- | ----------------- |
| 记录存在                               | 「锚点记录不存在」         |
| `case_result` 为 `failed` 或 `error` | 「锚点须为失败或异常用例」     |
| `analyzed` 为 `0` 或 `NULL`          | 「请选择未分析的失败记录作为锚点」 |


说明：要求锚点未分析，与「仅对未分析用例一键处理」一致，避免已分析记录作为入口造成误解。

---

## 5. 业务规则

### 5.1 用例开发责任人（跟踪人）解析

- 与列表接口中「用例开发责任人」规则一致：按 **`LOWER(main_module)` 与 `LOWER(ums_module_owner.module)`** 匹配（小写等价），再取 `for_reference` / `owner` 等字段。
- **姓名**：优先 `ums_module_owner.for_reference`；若为空则按 `ums_module_owner.owner`（工号）查 `ums_email.name`。
- **工号**：`ums_module_owner.owner`。
- **写入 `pipeline_failure_reason.owner` 的格式**：`**{姓名} {工号}`**（姓名与工号之间 **一个半角空格**）。若仅有工号或仅有姓名（极端情况），则与现有 `_format_case_dev_owner_display` 语义一致：能拼则拼，否则仅有的一侧写入。
- **长度**：`pipeline_failure_reason.owner` 为 `VARCHAR(100)`，若拼接后超长，规约：**截断至 100 字符**并打 **WARNING** 日志（或实现阶段二选一：拒绝该条并计入跳过，需在实现时与产品确认；**默认建议截断**以免阻塞整批）。

### 5.2 失败类型 `bug`

- 落库的 `failed_type` 须与 `**case_failed_type`** 表中代表 bug 的那条记录的 `**failed_reason_type` 值**一致（若库中为 `bug`/`Bug`，以实际存储为准；与 `spec/04` 中「忽略大小写识别 bug」的展示逻辑可并存，**写入值用字典真实值**）。

### 5.3 详细原因 `reason`（统一模板）

- 所有本批次成功写入的记录，`reason` **相同**，自动生成为：
  `**在{轮次}轮次新增失败`**
- 其中 `**{轮次}`** 为锚点（及本批）的 `**start_time` 字符串原样代入**（示例：`在202601201730轮次新增失败`）。
- 若 `start_time` 为空（不应出现在合法锚点中），实现应拒绝整单或跳过，不生成歧义文案。

### 5.4 分析人与分析时间

- `**analyzer`**：当前登录用户工号（与失败标注一致，来自 JWT `sub`）。
- **分析时间**：以 `**pipeline_failure_reason.updated_at`**（及 INSERT 时的创建时间）为准，**不使用**客户端上传时间。

### 5.5 `pipeline_history` 更新

- 对**成功写入** `pipeline_failure_reason` 的每条对应 `pipeline_history` 记录，将 `**analyzed` 置为 `1`**。

### 5.6 跳过规则（无匹配责任人）

- 若某条目标记录无法解析出用例开发责任人（例如 `**main_module` 为空**，或 `**ums_module_owner` 中无对应 `module`**）：**跳过该条**，不写入 `pfr`，不更新该条 `pipeline_history.analyzed`。
- 跳过须计入响应中的 **「跳过数量」**，并在日志中 **INFO** 记录（可含 `case_name`、platform，不含敏感信息）。

### 5.7 写入语义（INSERT / UPDATE）

- **不**复用「分析处理」的请求体或路由，但 `**pipeline_failure_reason` 与 `pipeline_history` 的写入键**与现网一致：按 `**(case_name, failed_batch, platform)`**，其中 `failed_batch` = `start_time`。
- 若该键已存在 `pfr`：执行 **UPDATE**（覆盖 `failed_type`、`owner`、`reason`、`analyzer` 等）；若不存在：**INSERT**。
- 说明：在「仅未分析参与」前提下，多数场景为首次写入；保留 upsert 与现网手动标注一致，避免脏数据无法纠正。

## 6. 与「继承」在 `analyzed` 上的一致性


| 项目                            | 继承（`spec/10`） | 一键分析（本文档） |
| ----------------------------- | ------------- | --------- |
| 是否仅处理 `analyzed = 0` 或 `NULL` | 是             | **是**     |
| 已分析记录                         | 不参与           | **不参与**   |


一键分析**不**要求继承规约中的「仅 INSERT pfr」不变量；一键分析采用第 5.7 节 upsert。

## 7. API 设计

### 7.1 请求


| 属性  | 规约                                                                            |
| --- | ----------------------------------------------------------------------------- |
| 方法  | `POST`                                                                        |
| 路径  | `**/api/v1/history/one-click-analyze`**（实现时可微调，但须独立路径，与 `failure-process` 区分） |
| 认证  | 需登录（与失败标注、继承一致）                                                               |


**请求体（JSON）**


| 字段                  | 类型    | 必填  | 说明                                                  |
| ------------------- | ----- | --- | --------------------------------------------------- |
| `anchor_history_id` | `int` | 是   | 锚点 `pipeline_history.id`，用于解析 `start_time` 及校验第 4 节 |


### 7.2 响应（建议）


| 字段                           | 类型       | 说明                                            |
| ---------------------------- | -------- | --------------------------------------------- |
| `success`                    | `bool`   | 是否整体调用成功（无未捕获异常）                              |
| `message`                    | `string` | 中文说明                                          |
| `batch`                      | `string` | 本批 `start_time`                               |
| `applied_count`              | `int`    | 成功写入并置 `analyzed=1` 的条数                       |
| `skipped_no_owner_count`     | `int`    | 因无责任人匹配而跳过的条数                                 |
| `skipped_not_eligible_count` | `int`    | 可选：因非 failed/error、已分析等被排除的条数（若后端统一在「候选集」上统计） |


当 `applied_count = 0` 且候选集中本应有未分析失败用例时，仍返回 **200**，`message` 说明「无成功条数」原因（例如全部无模块负责人）。

### 7.3 错误码（补充）


| HTTP  | 场景            |
| ----- | ------------- |
| `400` | 锚点校验失败（第 4 节） |
| `401` | 未登录           |
| `500` | 未预期异常         |


---

## 8. 前端行为（成功 / 失败）

- 成功：提示包含 **成功条数**、**因无责任人跳过条数**（若有）。
- 失败：展示后端 `detail` / `message`。
- **刷新表格**（与继承、手动标注成功后一致）。

---

## 9. 非功能与日志

- **日志**：成功 **INFO**（批次、`applied_count`、`skipped_no_owner_count`、操作者工号，跟踪人内容打码或省略）；校验失败 **WARNING**；异常 **ERROR** + `logger.exception`。
- **事务**：同一请求内对多行更新，建议**单事务提交**；单行失败策略为**跳过**而非整单回滚（与「跳过无责任人」一致）。

---

## 10. 实现检查清单（供开发自测）

- 仅处理 `start_time` 与锚点相同、且 `failed|error`、且 `analyzed` 为 0/NULL 的行。
- 不按 `subtask` 过滤批次内记录。
- `owner` 为 **姓名+工号** 格式写入 `pfr.owner`。
- `reason` 为 `**在{start_time}轮次新增失败`**。
- `failed_type` 与字典表 bug 一致。
- 无责任人匹配时跳过并计数。
- 新接口，**不**调用现有 `POST .../failure-process` 实现体（可复用内部纯函数，如责任人格式化、upsert 片段）。
