# 失败记录标注功能规约（Spec）

本文档在 `spec/02_history_fields_spec.md`、`spec/03_history_failure_reason_spec.md` 基础上，补充失败记录标注的交互与数据变更规约。

---

## 1. 功能概述

### 1.1 功能目标

针对详细执行历史页面的执行记录，支持用户批量或单条录入「失败类型」「跟踪人」「详细原因」等归因信息，并将数据持久化至 `pipeline_history` 与 `pipeline_failure_reason` 表，实现失败用例的标注闭环。

### 1.2 适用场景

- 测试人员或管理员对失败用例进行归因分析
- 批量标注同一失败原因的多条记录（如环境问题、已知 Bug）
- 单条精细标注（如指定模块、指定跟踪人的 Bug）

### 1.3 作用范围

- **页面**：详细执行历史页面
- **对象**：表格中的任意执行记录（不限于 `case_result = 'failed'`，但建议对非 failed 记录做限制或提示，见第 2 节）

---

## 2. 选择交互

### 2.1 表格行勾选

- 表格支持行勾选（Checkbox），每行左侧增加勾选框
- 勾选与「处理」按钮联动：**至少勾选 1 条记录**时，「处理」按钮可用；未勾选任何记录时，「处理」按钮禁用或置灰

### 2.2 全选能力

| 能力 | 是否支持 | 说明 |
|------|----------|------|
| 全选当前页 | 支持 | 表头 Checkbox 勾选时，选中当前分页内的所有记录 |
| 全选全部 | 不支持（或由产品决定） | 若支持，需明确与分页的关系：是否跨页全选；实现复杂度较高，建议首版仅支持「全选当前页」 |

### 2.3 对 case_result 的限制

- **允许对 `case_result = 'failed'` 或 `case_result = 'error'` 的记录进行标注**，原因：失败归因语义上针对失败用例与异常用例（如执行超时、环境异常等）
- **勾选行为**：
  - 方案 A（推荐）：勾选框仅对 `case_result = 'failed'` 或 `case_result = 'error'` 的行启用，passed 等行禁用勾选
  - 方案 B：允许勾选任意行，点击「处理」时过滤掉非 failed/error 记录，仅对 failed/error 记录弹窗；若全部为 passed，则提示「请至少选择一条失败/异常记录」
- **规约采用方案 A**：非 failed/error 行不显示勾选框或勾选框禁用，避免用户误选

### 2.4 与分页的关系

- 全选仅作用于当前页
- 切换分页后，上一页的勾选状态清空（或由产品决定是否跨页保持，建议清空以简化实现）

---

## 3. 入口与按钮

### 3.1 按钮位置

- 「处理」按钮放置在**表格上方工具栏**，与筛选条件区域同一行或相邻行，位于筛选控件右侧
- 与「刷新」等操作按钮并列，保持视觉层级一致

### 3.2 按钮状态

| 状态 | 条件 | 表现 |
|------|------|------|
| 可用 | 至少勾选 1 条 `case_result = 'failed'` 或 `'error'` 的记录 | 可点击，触发弹窗 |
| 禁用 | 未勾选任何记录，或勾选的全部为 passed | 置灰，不可点击 |
| 加载中 | 提交请求进行中 | 可显示 loading 状态（可选） |

### 3.3 与选择的联动

- 勾选/取消勾选时，实时更新按钮可用状态
- 点击「处理」后，弹窗展示，表格勾选状态可保持，便于用户取消后继续操作

## 4. 标注对话框

### 4.1 布局与结构

| 属性 | 规约 |
|------|------|
| 标题 | 「失败记录标注」 |
| 宽度 | 建议 480px～520px，适配表单项纵向排列 |
| 拖拽 | 可选，非必须 |
| 字段顺序 | 失败类型 → 模块（条件显示，仅 bug）→ 跟踪人 → 详细原因 |
| 底部按钮 | 左侧「取消」，右侧「确定」，符合 Ant Design Modal 默认布局 |

### 4.2 字段定义

| 字段 | 控件类型 | 必填 | 默认值逻辑 | 备选数据来源 | 联动逻辑 |
|------|----------|------|------------|--------------|----------|
| 失败类型 | 下拉单选 Select | 是 | 无默认，用户必选 | `case_failed_type.failed_reason_type`，按 id 或字母排序 | 选择 `bug` 时显示「模块」字段；切换失败类型时，跟踪人默认值联动更新 |
| 跟踪人 | **非 bug**：下拉 Select；**bug**：单行输入 Input | 是 | **非 bug**：选类型后默认 `case_failed_type.owner`（工号），选项来自 `ums_email`；**bug**：按当前模块匹配 `ums_module_owner`（`main_module` 与 `module` **忽略大小写**），将负责人工号在 `ums_email` 中解析为 **`姓名 工号`**（半角空格）填入输入框，**可改** | 同上 | 失败类型切换时重新取默认；**bug** 下切换模块时按新模块重算默认展示串 |
| 详细原因 | 多行文本框 TextArea | 是 | 无 | — | 无 |
| 模块 | 下拉单选 Select | 条件必填 | 默认 `pipeline_history.main_module`（见 5.3）；与选项列表 **忽略大小写** 匹配到规范模块名 | `ums_module_owner.module` | **仅当失败类型 = `bug` 时显示**；变更模块时跟踪人默认更新为对应 **`姓名 工号`** |

### 4.3 失败类型 `bug` 的匹配规则

- 匹配条件：`case_failed_type.failed_reason_type` 与字符串 `'bug'` 比较时，采用**忽略首尾空格、大小写不敏感**的规则
- 即：`'bug'`、`'Bug'`、`' BUG '` 等均视为 `bug`，触发模块字段显示

### 4.4 校验规则

| 字段 | 校验规则 |
|------|----------|
| 失败类型 | 必选，未选择时提示「请选择失败类型」 |
| 跟踪人 | 必填；非空校验提示「请填写跟踪人」；**bug** 时建议最大长度 100 字符（与 `pipeline_failure_reason.owner` 一致） |
| 详细原因 | 必填，空或仅空格时提示「请输入详细原因」；建议最大长度 2000 字符（与 `reason` text 类型兼容） |
| 模块 | 当失败类型为 `bug` 时必选，未选择时提示「请选择模块」 |

### 4.5 确定/取消行为

| 操作 | 行为 |
|------|------|
| 取消 | 关闭对话框，不发起任何请求，不修改数据；表格勾选状态可保持 |
| 确定 | 1）前端校验必填项；2）校验通过后调用后端 API 提交；3）提交成功后关闭对话框，刷新表格数据；4）提交失败则展示错误信息，对话框不关闭，允许用户修改后重试 |

## 5. 数据变更规约

### 5.1 pipeline_history

| 字段 | 更新规则 | 匹配条件 |
|------|----------|----------|
| analyzed | 置为 1 | 对每条选中的 `pipeline_history` 记录，按 `id` 更新 |

### 5.2 pipeline_failure_reason

| 操作 | 判断逻辑 | 匹配键 | 写入字段 |
|------|----------|--------|----------|
| INSERT | `pipeline_failure_reason` 中不存在 `(case_name, failed_batch, platform)` 匹配记录 | `case_name`、`failed_batch`（= `pipeline_history.start_time`）、`platform` | `owner`、`reason`、`failed_type`、`analyzer`（当前操作人，见下）、`created_at`（数据库默认）、`updated_at`（数据库默认） |
| UPDATE | 已存在匹配记录 | 同上 | `owner`、`reason`、`failed_type`、`analyzer`、`updated_at`（数据库 ON UPDATE） |

**analyzer 字段**：写入当前登录用户标识（工号或姓名，由系统登录态决定），表示「谁进行了本次标注」。

### 5.3 批量处理策略

- **多条记录共用同一套表单值**：用户勾选多条记录后，点击「处理」，弹出一个对话框，表单内填写一套值（失败类型、跟踪人、详细原因、模块），点击「确定」后，**对所有勾选记录应用同一套值**
- **每条记录独立写入**：每条 `pipeline_history` 记录对应 0 或 1 条 `pipeline_failure_reason`，按 `(case_name, start_time, platform)` 逐条判断 INSERT 或 UPDATE

### 5.4 模块字段的批量默认值

- 当失败类型为 `bug` 且勾选多条记录时，模块默认值策略：
  - **方案 A**：取勾选记录中 `main_module` 的众数（出现次数最多的值）；若众数不唯一，取第一条记录的 `main_module`
  - **方案 B**：取第一条勾选记录的 `main_module`
- **规约采用方案 B**：简单可预期，实现成本低

### 5.5 API 设计要点

**请求**

- 方法：`POST`
- 路径：`/api/v1/history/failure-process`（或等价路径）
- 请求体结构：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| history_ids | array[int] | 是 | 选中的 `pipeline_history.id` 列表 |
| failed_type | string | 是 | 失败类型，来自 `case_failed_type.failed_reason_type` |
| owner | string | 是 | **非 bug** 多为工号；**bug** 时为失败跟踪人展示串（如 **`姓名 工号`**），与 `pipeline_failure_reason.owner` 一致，可自定义 |
| reason | string | 是 | 详细失败原因 |
| module | string | 条件 | 当 `failed_type` 为 `bug` 时必填，来自 `ums_module_owner.module` |

**响应**

- 成功：`200`，body 可返回 `{ "success": true, "message": "标注成功" }` 或简化结构
- 失败：`4xx`/`5xx`，body 包含 `message` 或 `detail` 供前端展示

**后端校验**

- 校验 `history_ids` 对应的记录存在且 `case_result` 为 `'failed'` 或 `'error'`（防止篡改请求）
- 校验 `failed_type`、`owner`、`reason` 非空；当 `failed_type` 为 `bug` 时校验 `module` 非空

## 6. 边界与异常

### 6.1 部分有记录、部分无记录

- 勾选的多条记录中，部分已有 `pipeline_failure_reason`、部分没有，**统一按逐条处理**：
  - 有则 UPDATE，无则 INSERT
  - 无需额外提示，用户无感知

### 6.2 提交失败

- 网络错误、超时：提示「网络异常，请稍后重试」
- 业务错误（如记录不存在、权限不足）：展示后端返回的 `message` 或 `detail`
- 对话框保持打开，允许用户修改后再次点击「确定」

### 6.3 提交成功后刷新策略

- **推荐**：重新拉取当前页数据（保持分页、筛选条件不变），使表格中的 `analyzed`、`failure_owner`、`failed_type` 等列立即更新
- 若当前有 Drawer 打开且展示的是已标注记录，Drawer 内容应同步更新（可依赖表格数据刷新后重新渲染）

### 6.4 并发与幂等

- 若用户快速连续点击「确定」，建议前端防抖或禁用按钮，避免重复提交
- 后端对同一批 `history_ids` 的重复请求，可视为幂等：相同参数多次执行，结果一致（UPDATE 覆盖）

## 7. 与现有规约的衔接

### 7.1 表格列更新

- 标注完成后，表格中的「是否已分析」「跟踪人」「失败原因」列（来自 `spec/02_history_fields_spec.md`、`spec/03_history_failure_reason_spec.md`）应**立即更新**
- 通过刷新当前页数据实现，与 6.3 一致

### 7.2 Drawer 失败归因区更新

- 若用户标注后，Drawer 正在展示该条记录，失败归因区的「跟踪人」「失败原因」「详细原因」「分析人」「分析时间」应同步更新
- 实现方式：表格数据刷新后，Drawer 展示的数据源来自表格行或同一接口，自然联动

### 7.3 筛选与 URL

- 标注不改变筛选条件与 URL 查询参数
- 用户若筛选了「跟踪人」「失败原因」，标注后新写入的数据若符合筛选条件，会出现在当前结果集中；若不符合，可能因刷新而移出视野，属预期行为

### 7.4 分析处理记录（预留）

- Drawer 中计划在失败归因区下方新增「分析处理记录」子块，用于展示同一执行记录的历史分析处理记录（如多次标注的记录列表）。
- 展示形式（如时间线、折叠卡片、列表等）以及与后端数据结构的具体对应关系将在后续 Spec 中补充，本规约仅做占位说明，提醒在实现与设计阶段预留相应扩展空间。

## 8. 附录：数据流示意

```
用户勾选 failed 记录 → 点击「处理」→ 弹窗填写表单
    → 确定 → 前端校验 → POST /api/v1/history/failure-process
    → 后端：遍历 history_ids
        → 更新 pipeline_history.analyzed = 1
        → 按 (case_name, start_time, platform) 判断
            → 无 pfr 记录：INSERT pipeline_failure_reason
            → 有 pfr 记录：UPDATE pipeline_failure_reason
    → 返回成功 → 前端刷新表格 → 关闭弹窗
```
